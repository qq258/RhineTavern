# 莱茵酒馆 (Rhine Tavern) - 后端底座技术规格书 (Feature Spec)

## 1. 架构概述
本项目采用前后端分离架构，基于 Turborepo 构建的 Monorepo 工程。后端底座旨在为“微信小程序（敏捷录入端）”与“Vue 3 Web 后台（数据管理端）”提供统一、安全、高可用的 API 服务。底层数据存储采用 PostgreSQL，并通过 Prisma ORM 进行模型映射与数据访问。

## 2. 全局工程规范

### 2.1 API 版本控制策略
* **无感升级:** 路由层不使用 `/v1/` 等硬编码 URL 前缀。
* **Header 驱动:** 客户端请求必须携带 `X-API-Version` Header（默认 `1.0.0`）。后端网关/中间件通过读取该 Header 决定路由分发或下发升级提示。

### 2.2 时区与时间处理
* **存储标准:** 所有业务时间字段（如 `recordDate`, `createDate` 等）在入库前强制转换为 **UTC 时间** 存储。
* **业务时区:** 系统业务逻辑默认绑定为 **北京时间 (UTC+8)**。客户端传入时间必须携带时区标识，后端统一解析抹平差异。

### 2.3 数据删除规范 (Soft Delete)
* **全局软删除:** 数据库层面禁止物理删除（级联删除 `sys_user_roles` 等纯中间映射表除外）。
* **底层支撑:** 所有实体表均已包含 `deleted` (Boolean) 和 `deletedDate` (DateTime) 字段。
* **查询约束:** 后端所有数据列表查询与详情查询，默认追加 `where: { deleted: false }` 条件。

## 3. 中间件与防护机制设计

### 3.1 鉴权与 Token 策略 (Auth Middleware)
后端实现双端 Token 校验策略：
* **后台管理端:** Token 有效期为 **1 小时**。过期拦截后返回 `401 Unauthorized`，要求强制重新登录。
* **微信小程序端:** Token 有效期为 **1 天**。配合小程序静默逻辑实现无感刷新，保障健身体验连续性。
* **越界防护:** 配合 RBAC 权限表，跨权访问拦截并记录至安全日志，返回 `403 Forbidden`。

### 3.2 流量与性能熔断 (Circuit Breaker Middleware)
* **分页限制:** 常规列表分页默认 `limit = 10`，硬性上限锁定为 `limit = 50`。
* **全量拉取熔断:** 针对食物库、动作库的初始化拉取接口，后端强制设定 **2000 条** 数据红线。触发红线立即阻断请求，并下发特定状态码 `42901` (Payload Too Large / Switch to Pagination)，强制前端降级为分页查询。

### 3.3 幂等性与防抖 (Idempotency)
* **请求指纹:** 核心写操作（如 `workout_sets` 训练打卡写入）需验证客户端生成的 UUID 请求标识，配合后端 Redis 或内存锁，防止弱网环境下的重试导致数据重复生成。

## 4. 核心业务规则引擎

### 4.1 历史数据快照不可变规则 (Immutable Snapshots)
* **规则定义:** 历史打卡数据中的营养素（总热量、总碳水等）一旦落盘，绝对不可追溯修改。
* **逻辑闭环:** 即使后台管理员在 `foods` 表中修正了某项食物的百克热量，该修正**仅对修改发生时间之后**的新增饮食记录生效，确保用户的历史卡路里缺口数据永久准确。

### 4.2 训练记录可修改性与时间窗口 (Workout Edit Window)
* **7 天编辑窗口:** 允许用户自由修改过去 7 天内的 `workout_logs` 及关联 `workout_sets`。超过 7 天的数据从 API 层面被锁定为只读。
* **草稿回填机制 (UI/UX 联动):** * 用户对过期错误数据执行“删除重建”时，后端不仅执行软删除，同时在接口中返回被软删除实体的完整 JSON 结构。
    * 该结构作为“临时草稿”供小程序前端回填至表单（结合 Pinia 状态管理），用户可手动调整错误字段（如手误多敲的一个零）后作为新记录重新入库，保留原始记录发生时间。

## 5. API 响应契约 (Response Schema)

为保证前端 Axios 拦截器统一处理，所有 API 强制遵循以下 JSON 响应结构：

**成功响应 (200 OK):**
```json
{
  "success": true,
  "code": 20000,
  "message": "操作成功",
  "data": { ... } // 实体对象或分页包裹对象
}
```

**业务异常响应 (如 400 Bad Request / 42901 熔断):**
```json
{
  "success": false,
  "code": 42901,
  "message": "全量数据体积过大，请切换至分页加载模式",
  "data": null,
  "traceId": "req-uuid-12345" // 用于日志追踪
}
```
