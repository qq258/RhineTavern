# 莱茵酒馆 (Rhine Tavern) - 后端 API 开发指令书 (AI 助手执行指南)

## 0. 背景与系统定位
你现在是该项目的资深 Node.js 开发工程师。本项目是一个名为“莱茵酒馆”的个人健身与健康管理系统，包含微信小程序端（记录）和 Vue3 后台管理端（分析）。
当前工作区：`apps/backend`
技术栈：Node.js + Express + TypeScript + Prisma ORM + PostgreSQL。
数据库模型 (`schema.prisma`) 已就绪，所有实体表均包含审计字段和 `deleted`, `deletedDate` 字段以支持软删除。

---

## 1. 严格工程规范 (全局必须遵守)

在编写任何代码之前，请绝对遵循以下架构约束：
1. **统一响应格式:** 所有 API 成功响应必须包裹为 `{ "success": true, "code": 20000, "message": "操作成功", "data": ... }`。
2. **全局软删除:** 绝对禁止调用原生的 `prisma.xxx.delete` 或 `deleteMany`。所有的删除操作必须转化为 `update({ data: { deleted: true, deletedDate: new Date() } })`。请在 Prisma Client 初始化时使用 `Prisma.defineExtension` 编写全局拦截器。
3. **查询过滤:** 所有的 `findMany`, `findFirst`, `findUnique` 查询，必须默认带上 `where: { deleted: false }`，同样通过 Prisma 扩展拦截实现。
4. **时间标准:** 所有入库时间强制转换为 UTC，不处理时区偏移。
5. **快照不可变原则:** 饮食打卡 (`diet_logs`) 写入时，必须实时读取 `foods` 的热量/营养素并计算绝对值落库。后续不允许任何同步更新历史数据的操作。
6. **7 天编辑窗口:** 对 `workout_logs` 的修改与删除，必须校验 `recordDate`，超过 7 天 (168小时) 的记录拒绝操作并抛出业务异常。

## 2. 目录结构规范
请按照以下结构在 `apps/backend/src` 下组织代码：
```text
src/
├── config/           # env 变量、Prisma 客户端实例化 (包含软删除扩展)
├── middlewares/      # auth.ts, response.ts (统一返回体), error.ts (全局异常), version.ts
├── controllers/      # 解析参数，调用 service，返回统一格式
├── services/         # 核心业务逻辑 (事务、规则校验)
├── routes/           # 路由定义 (index.ts 汇总)
├── utils/            # 通用工具
└── index.ts          # Express App 入口及中间件挂载
```

## 3. 开发执行步骤 (分阶段执行)

### 【阶段一：基础设施与中间件建设】
**目标：搭建 Express + TS 骨架，实现核心拦截器。**
1. 初始化 Express 服务 (`src/index.ts`)，配置 CORS 和 Body Parser。
2. 实例化 Prisma Client (`src/config/prisma.ts`)，**必须编写并挂载全局软删除及查询过滤 Extension**。
3. 编写统一响应格式中间件 (`src/middlewares/response.ts`)，挂载到 `res.success = (data) => {...}`。
4. 编写全局异常捕获中间件 (`src/middlewares/error.ts`)，捕获自定义 `BusinessError`，输出标准化 JSON 错误信息。
5. 编写 API 版本控制中间件 (`src/middlewares/version.ts`)，校验请求头 `X-API-Version`。

### 【阶段二：鉴权体系搭建】
**目标：实现基于 JWT 和 RBAC 的安全防护。**
1. 编写鉴权中间件 (`src/middlewares/auth.ts`)：提取 Bearer Token，校验有效性，区分后台 (1小时) 和小程序 (1天) 的过期策略。
2. 实现 `AuthService` 和 `AuthController`：包含登录颁发 JWT 的逻辑。

### 【阶段三：核心业务逻辑 (Services & Controllers)】
**目标：实现业务闭环，严格执行业务规则。**
1. **基础字典模块 (Food/Exercise):**
   * 实现 CRUD。
   * **熔断机制:** 在 `findMany` 之前校验，如果前端未传分页参数且数据总量 `count > 2000`，抛出 `42901` 异常。
2. **饮食打卡模块 (Diet):**
   * 实现打卡录入，严格遵守“实时计算、快照不可变”原则。
3. **训练记录模块 (Workout):**
   * 实现打卡写入。
   * **更新/删除校验:** 严格执行 7 天时间窗口校验。
   * **草稿回填机制:** 提供带有 `?returnDraft=true` 参数的 DELETE 接口。在执行软删除后，将该 `workout_logs` 及其关联的 `workout_sets` 的完整 JSON 树通过 `data` 返回，供前端回填。

### 【阶段四：路由组装与联调支持】
**目标：对外暴露 API 并进行测试。**
1. 在 `src/routes/` 目录下按照模块划分路由文件。
2. 统一在 `src/routes/index.ts` 中注册，并挂载鉴权中间件。
3. 确保可通过 `npm run dev` 正常启动，并能处理跨域请求。
