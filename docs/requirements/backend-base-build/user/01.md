1. **边界条件应答**

- **时区与日期边界**：
  - 在本项目中，由于面向用户只包含国内用户，所以时区是固定的，使用北京时间作为默认时区。
  - 如果确实出现了时区差异，我倾向于使用强制要求所有入参都带时区然后统一转为UTC存储。

- **分页与全量拉取**:
  - 查询接口提供两种，一种是分页查询，一种是全量拉取。
  - 分页查询默认每页10条，最大每页50条，但是可以通过一个开关来控制是否可以突破最大每页50条的限制。
  - 全量拉取时，返回所有数据，在产出接口文档时进行标注，提示谨慎使用。

2. **异常流应答**

- **Token续期与失效**:
  - token分为后台管理系统和小程序两种，如果是后台管理系统，token过期时间为1小时，小程序token过期时间为1天。
  - 如果token过期，后台管理系统用户需要重新登录获取新的token，小程序用户可以无感刷新。

- **权限越界拦截**:
  - 需要记录安全日志，记录用户操作、权限越界信息等。

- **防抖与幂等性**:
  - 小程序在点击按钮发送请求时，会添加防抖机制，防止重复点击导致重复请求。
  - 同时，接口会添加幂等性处理，确保重复请求不会对数据造成影响。

3. **状态流转与业务约束应答**

- **历史数据的修改权限**:
  - 分情况限制，例如：
    - 对于食物、食谱等历史数据，用户只能在自己创建后修改自己的数据。
    - 对于健身记录等数据，用户不能修改。

- **关联数据的级联删除危机**:
  - 采用软删除策略，删除关联数据时，只更新关联数据的删除状态为已删除，而不是物理删除。

4. **历史兼容应答**

- **API 版本控制**:
  - 采用版本号管理，每个接口都有一个版本号，版本号默认为1.0.0。
  - 当接口有重大变更时，会增加版本号，例如1.0.1。
  - 小程序在调用接口时，需要将本地的版本号放到请求头中，接口会根据版本号判断是否需要升级接口。
  - 如果版本号有变更，小程序会提示用户升级接口。
  - 不需要强制引入前缀版本隔离，因为版本号已经包含了接口的变更信息。

- **快照数据的逻辑闭环**:
  - 快照字段实时计算，如果修改了原始数据，快照字段会自动更新。比如修改了 food 的热量，那么对应的 totalKal 也会自动更新。
  - 针对这种关联修改的情况，在修改原始数据时，后端需要记录改动日志，记录修改的字段、修改前后的值、修改人等信息。
