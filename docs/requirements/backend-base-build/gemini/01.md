这份“后端底座搭建”的需求虽然只有一句话，但作为整个“莱茵酒馆”系统的地基，中间件和路由的设计将直接决定后续开发的顺畅度与系统的健壮性。

既然你的 Prisma Schema 已经落库，我仔细审视了表结构与基础请求链路，发现如果不提前把以下这些业务规则和工程细节敲定，后续极大概率会面临重构。

请从以下四个维度回答我的灵魂拷问：

### 1. 边界条件 (Boundary Conditions)
* **时区与日期边界：** 我们的核心表（如 `daily_metrics`、`diet_logs`、`workout_logs`）重度依赖 `recordDate`。由于系统包含 PC 后台和微信小程序，如果客户端处于不同时区（或者跨越了零点），后端的“今天”该如何界定？我们是强制要求所有入参都带时区的 ISO 字符串，然后统一转为 UTC 存储，还是在中间件层直接抹平时区差异？
* **分页与全量拉取：** 在获取 `foods` 和 `exercises` 这些基础数据源时，小程序端可能会为了体验流畅而要求“下拉触底加载”或“全量缓存”。后端的查询接口是否有强制的最大 `limit` 限制（例如单次请求最多 100 条）以防止把数据库或内存打挂？

### 2. 异常流 (Exception Flows)
* **Token 续期与失效：** 中间件必然要处理 JWT 校验。如果 Token 过期了，你的期望是直接扔回 `401 Unauthorized` 让小程序无情地弹回登录页，还是希望后端搭建一套双 Token（Access + Refresh）的无感刷新机制？
* **权限越界拦截：** 基于 RBAC 模型，假设普通用户（或黑客）拦截请求，尝试调用只属于 Admin 的 `DELETE /api/foods/:id` 接口，鉴权中间件是应该静默忽略、返回 `403 Forbidden`，还是记录一次安全告警日志？
* **防抖与幂等性：** 健身房网络往往很差，如果用户在小程序上疯狂点击“完成本组训练”（写入 `workout_sets`），路由层或业务层是否有幂等性校验机制，以防止在同一秒内生成两条完全一样的 `set_index` 数据？

### 3. 状态流转与业务约束 (State Transitions)
* **历史数据的修改权限：** 健身数据录入后，状态是“可变”的吗？比如，用户能在周五去修改周一的 `diet_logs` 或 `daily_metrics` 吗？如果可以，有没有时间窗口限制（比如只能修改近 7 天的数据）？
* **关联数据的级联删除危机：** 这是一个致命的架构问题。在 Prisma 结构中，`diet_logs` 关联了 `foods` 表，`workout_sets` 关联了 `exercises` 表。如果后台管理员因为发现某个食物名字建错了，直接**删除了**这个 Food，此时那些已经引用了这个 `foodId` 的用户的历史饮食记录该怎么办？是直接报错阻止删除、自动级联删除用户的记录，还是采用“软删除”（增加 `isDeleted` 字段）？

### 4. 历史兼容 (Historical Compatibility)
* **API 版本控制：** 小程序发版是有审核周期的，这意味着用户手机上可能会存在旧版本，而后端可能已经上线了新代码。路由设计上，我们是否从第一天起就强制引入 `/api/v1/` 这样的前缀进行版本隔离？
* **快照数据的逻辑闭环：** 我注意到你在 `diet_logs` 中设计了 `totalKcal` 等冗余字段作为快照，这非常棒，解决了食物库数值更新导致历史热量突变的问题。但是，如果管理员修改了 `foods` 里的基础热量，后端是否需要提供一个“历史数据重新计算”的刷数接口，还是说老数据绝对不可篡改，新热量只对未来的打卡生效？

想清楚这些问题，我们的“底座”才算真正稳固。请告诉我你对这些场景的倾向和决策。
