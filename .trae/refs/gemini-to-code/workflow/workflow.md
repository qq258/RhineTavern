# 使用 gemini 结合 trae 实现需求落地的全流程

### Phase 0: 需求反刍与澄清
在这一步，你需要将收集到的需求放到 `../gemini-tech-leader.md` 中，然后调用 gemini 进行漏洞扫描。

### Phase 0.5: 迭代与对齐（双脑对话）
1. **gemini提出问题** : ` 我会根据你的需求，列出比如：“跟进记录能删除吗？”、“跟进时间是系统自动取当前时间，还是允许业务员手动回溯修改？” 等极其具体的问题。 `
2. **你进行解答或回退**:
  - 如果你能做主，你直接在对话框里回答我：“不允许删除，只能追加；时间取系统当前时间。”
  - 如果你发现这确实是产品经理没想清楚的，你就可以拿着这些问题去“拷问”你的产品经理了。
3. **循环直至需求清晰**: 和 gemini 反复对话，直到你觉得所有业务逻辑都天衣无缝了。

### Phase 1: 降维与输出规格书
当把需求讨论透彻后，只需要对gemini说出终极指令:
`“需求已经明确，现在请切换到‘资深全栈架构师’角色。结合我们刚才讨论补充的所有细节，按照《功能技术规格书 (Feature Spec)》的 Markdown 模板，为我输出最终的技术方案。”`

### Phase 2: 领域模型与契约落地
拿着刚才生成的 Markdown 文档，回到你的 Trae 中开始执行。
1. **更新 Prisma (后端)**：让 Trae 读取文档中的数据模型部分，去修改 `apps/backend/prisma/schema.prisma`。
- *执行*：修改完成后，运行 `npx prisma db push` 和 `npx prisma generate`。
3. **定义共享类型 (Shared)**：这一步是 Monorepo 的灵魂！让 Trae 根据文档中的 API 契约，在 `packages/shared/src/types` 下生成 `TypeScript` 的 interface。
- *效果*：这样前端和后端立刻就拥有了完全一致的类型推导，彻底消灭“字段拼写错误”导致的联调 Bug。

### Phase 3: 后端业务流开发
底层数据和类型都有了，现在开始写业务逻辑。
1. **生成 Controller / Service**：让 Trae 读取 `feature-followup-record.md`，并在 `apps/backend` 中生成对应的 Express 路由和处理函数。
2. **绑定 Prisma 与 Shared**：Trae 会自动引入 `packages/shared` 中的接口定义来校验入参，并调用 Prisma Client 操作数据库。

### Phase 4: 前端视图与联调
后端接口 Ready（或至少定义好类型）后，进入前端环节。
1. **生成 API 请求层**：让 Trae 在 `apps/frontend/src/api/` 下生成调用该接口的 axios 函数，并引入 `packages/shared` 的类型作为返回值。
2. **搭建 Vue3 视图**：让 Trae 根据 Markdown 中的组件树设计，生成 `.vue` 文件。由于数据模型和接口结构都已经非常清晰，AI 生成的 Vue 模板数据绑定会极其精准。
3. **联调微调**：在浏览器中查看效果，遇到样式不对或小 Bug，直接在 Trae 中框选代码让其修复。

---

**注意：Phase 2,3,4 需要分步执行，不能直接告诉 trae 一次性执行所有步骤。**
### 💡 为什么这种“挤牙膏”式的方法最高效？
1. **Bug 隔离**：如果 Trae 在写 Prisma Schema 时把字段名拼错了，你在第一步就能立刻发现并让它纠正。如果让它一次性全写完，前端代码可能已经用着那个错误的字段名写了几百行了，改起来极其痛苦。

2. **利用 Monorepo 的类型红利**：第二步和第三步完全依赖于第一步生成的 packages/shared 里的 TypeScript 接口。分步执行能确保 AI 在写业务逻辑时，有一个强类型的“护栏”约束着它，极大降低了运行时报错的概率。

3. **极低的心智负担**：你就像在监工，看着进度条 33% -> 66% -> 100% 稳步推进，心里非常有底。
